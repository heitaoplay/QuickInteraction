# Echo 数据迁移后清理原插件数据 — 可行性评估

> 状态：纯评估，未实现。
> 背景：原 echo 插件（echo-activity-ext）的自定义动作功能已过时，计划把数据迁移到本插件。迁移完成后原 echo 不再使用，建议在迁移后清理原 echo 插件里的自定义动作数据，仅保留本插件数据。删除由用户自行决定，本插件提供建议与提示。

---

## 0. 与上一轮「清空旧数据」的本质区别（重要）

| | 上一轮方案 | 本轮方案 |
|---|---|---|
| 删除目标 | 本插件内 `source==='echo'` 的动作 | **原 echo 插件** `Player.ExtensionSettings[ECHO]['动作数据']` |
| 数据归属 | 我们自己的存储 | 外部插件的数据 |
| 风险性质 | 误删 native、丢本地编辑 | 改动外部插件状态（可控） |
| 结论 | 不推荐 | **可行** |

本轮删除的是"别人家"的冗余数据，我们自己的数据完好无损，因此上一轮的高危误删、不可恢复等风险**不再成立**。

---

## 1. 当前迁移能力（已具备）

`importCustomFromEcho()`（src/08-custom-d.js:11）已经实现了"从 echo 读 → 转成本插件格式 → 存进 `state.customActions`"的完整迁移逻辑：
- 读取 `Player.ExtensionSettings` 里 `ECHO*` 键下的 `动作数据`
- 逐个解析 displayName / group / scope / dialog / dialogSelf，归一化占位符
- 记录 `echoName` / `echoNames`（echo 原始 Activity 名，用于屏蔽）
- 覆盖式去重后 `upsertCustom` 进我们存储

**所以"导出并迁移"这一步已经存在**，无需新建"导出"流程——我们代码内直接读 echo 数据即完成迁移。

## 2. 清理原 echo 数据后，屏蔽机制是否仍有效？

核心问题：迁移后我们依赖 `echoSuppressed` 集合 + `caRemoveSuppressedEchoActivities()` 屏蔽 echo 原始 Activity。如果清空了原 echo 数据，`rebuildEchoSuppressed()` 在启动时会调用 `caGetEchoData()` 扫描 echo 数据补益屏蔽名——这步将失效。

但屏蔽的真正来源是 **`state.customActions[].echoName / echoNames`**（已持久化到我们存储），不是 echo 数据本身。`rebuildEchoSuppressed` 优先用这些字段（src/05-custom-a.js:252-253），echo 数据只是"补充发现"。

**结论**：清空原 echo 数据后，只要 `echoName`/`echoNames` 在我们这边完整，屏蔽机制**仍然有效**。系统反而更自包含（不再依赖外部 echo 插件的数据）。

若用户同时**卸载 echo 插件**，其原始 Activity 根本不再注册，最干净，无任何冲突源。

---

## 3. 可行性结论

✅ **方案可行，且比"保留双份"更优**：
- 数据已完整迁移到我们这边，原 echo 数据成为冗余
- 原 echo 插件若仍加载，每次启动会重新向 BC 注册表注册它的原始 Activity（`笨蛋笨Luzi_xw58d`），我们仍要靠屏蔽集合去盖掉——这是持续的运行负担与潜在竞争
- 清理后：屏蔽集合仅靠我们自己的 `echoName` 即可，启动扫描不再依赖外部数据，更稳定

---

## 4. 推荐实现路径（更优方案）

### 4.1 迁移入口（复用现有）
保留 `importCustomFromEcho` 作为"迁移"手段，无需新建导出流程。

### 4.2 迁移后检测 + 建议提示（核心新增）
导入成功后：
1. 检测 `caGetEchoData()` 是否仍存在数据
2. 若存在 → 显示**非阻断建议提示**（toast / 可关闭 banner）：
   > "已从 echo 迁移 N 个动作。原 echo 插件的动作数据仍可清理，建议在确认新插件运行正常后删除，系统将更轻量稳定。"
3. 提供可选按钮「清理原 echo 数据」+「暂不处理」

### 4.3 「一键清理」的安全做法
点「清理原 echo 数据」后，执行：
- **只清空 `Player.ExtensionSettings[ECHO]['动作数据']` 这一个字段**（设为 `{}` 或删除该键），**不动 echo 插件的其他配置**，降低副作用
- 立即 `rebuildEchoSuppressed()` 基于我们的 `echoName` 重建屏蔽，确保屏蔽不丢
- `toast` 提示完成
- 可选：清理前自动把原 echo 数据导出一份 JSON 备份到本地存储（或下载），满足"可逆"

### 4.4 不代删、只建议的保守变体
若担心动外部插件数据，可改为：提示中给出"请到 echo 插件设置中删除动作数据"的指引，本插件不实际修改 echo 数据。这最符合"删除由用户自行决定"，但体验略弱（需用户手动操作）。

---

## 5. 风险与缓解

| 风险 | 可能性 | 缓解 |
|---|---|---|
| 清空 `动作数据` 字段导致 echo 插件启动报错 | 低（多数插件读空数据安全）；取决于 echo 实现 | 只清该字段不动其他；建议用户同时卸载 echo 插件 |
| 清空后屏蔽失效 | 无（屏蔽靠我们的 `echoName`） | 清理后立即 `rebuildEchoSuppressed` |
| 用户清理后后悔 | 低（我们数据已完整） | 清理前自动备份 JSON；且我们数据独立，可恢复 |
| 多次迁移导致提示反复弹 | 中 | 用 `state.echoMigrated` 持久化标记，已迁移且原数据已空则不再提示 |

上一轮分析的是"导入时清空**本插件** echo 动作"——结论是不推荐（误删 native、丢编辑）。本轮是"迁移后清理**原 echo 插件**数据"——结论是推荐。两者删除目标不同，结论相反，请勿混淆。
