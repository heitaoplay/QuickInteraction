# 导入 Echo 时「清空旧 Echo 数据」方案可行性分析

> 状态：纯分析，未实现。
> 背景：用户提议在点击「从 echo/回声 导入」时，先弹确认框询问是否删除当前已有的 Echo 动作数据，通过清除旧数据来避免新旧动作之间的冲突。

---

## 0. 先厘清：当前已有的冲突处理机制

`importCustomFromEcho()`（src/08-custom-d.js:11）当前并非"简单追加"，而是已经内置了两层处理：

1. **覆盖式更新去重**（第 44–55 行）：按 `(displayName, group)` 查找现有动作，已存在则**原地更新** dialog / scope / echoName 等字段，保留原 `id`、`createdAt`；不存在才新增。
2. **屏蔽 echo 原始 Activity**（第 72–83 行）：把 echo 原始 Activity 名（如 `笨蛋笨Luzi_xw58d`）写入 `state.echoSuppressed` 并调用 `caRemoveSuppressedEchoActivities()`，从 BC 注册表物理移除，避免面板出现"本插件 XSQAct_ 版 + echo 原始版"两份同名动作。

换言之，"重复导入导致面板出现双份动作"这个问题，**当前设计已经通过更新去重 + 屏蔽在大部分情况下规避了**。

### 真正可能残留的"冲突"场景

| 场景 | 成因 | 当前是否被覆盖 |
|---|---|---|
| A. echo 插件升级后，同一中文名对应的原始 Activity 名变了 | echo 注册名带随机后缀，升级后从 `_xw58d` 变 `_abc12` | 部分：去重会更新 `echoName`，但旧的屏蔽名可能残留（启动时 `rebuildEchoSuppressed` 会重新扫描修正，多数情况自愈） |
| B. 用户对导入的 echo 动作做了**本地二次编辑**，再导入被覆盖 | 去重逻辑直接覆盖 dialog 等字段 | 否：用户编辑会丢失 |
| C. echo 把某动作换了 group / 关键字段 | 去重按 `(name, group)` 匹配，group 变了就 miss → 新增而非更新 | 否：会出现"旧 group 一份 + 新 group 一份"的重复 |

因此"冲突"主要指 **B（本地编辑丢失）** 和 **C（字段变化的重复）**，以及少量 A 的边界残留。

---

## 1. 该方案（清空旧数据再重导）是否能有效解决冲突？

- **对 C 有效**：全删重导，不会有旧 group 的遗留版本。
- **对 A 部分有效**：删掉旧 echo 动作后，依赖 `rebuildEchoSuppressed()` 在启动时基于现存 echo 动作重建屏蔽集合，若旧动作已删、新动作已导入，重建结果干净。但需注意：**仅删 `state.customActions` 不够，必须同步清空 `state.echoSuppressed` 并对每个被删动作调用 `caUnregister`**，否则旧的 XSQAct_ Activity 与屏蔽名会残留。
- **对 B 无效，反而有害**：清空意味着用户所有基于 echo 导入后的二次编辑**全部丢弃**——这比现有"覆盖更新"更激进（覆盖只丢 dialog 等几个字段，清空连可见性、排序、改名都丢）。

**结论：能解决"重复"类冲突，但代价是顺带消灭了所有本地个性化，且对"本地编辑丢失"这一核心痛点毫无改善。**

---

## 2. 风险与负面影响

### 2.1 误删用户自己的 native 动作（高危）
`importCustomFromEcho` 第 45–51 行的去重**不区分 source**：若用户在「我的动作」里手动创建了一个叫"亲亲"（ItemMouth）的 native 动作，再导入 echo 的"亲亲"（ItemMouth），会**命中并更新这个 native 动作，且把它的 `source` 改写成 `'echo'`**（第 51 行）。

后果：该用户原生动作已经被"同化"为 echo source。此时用户点"删除所有 Echo 数据"，会**把这个用户自己创建的 native 动作一并删掉**。这是该方案最隐蔽、最严重的副作用。

### 2.2 数据不可恢复
删除即清除，无撤销。误删后只能重新从 echo 导入，而若用户曾在本地改过对话词、可见性、排序，这些信息彻底消失。

### 2.3 排序与可见性丢失
用户在「我的动作」里拖拽排好的顺序、手动隐藏的某些 echo 动作，清空后全部重置（重导按 echo 原始顺序）。

### 2.4 实现若不完整会引入新 bug
"清空"必须严格做到三件事，缺一不可：
1. 遍历所有 `source === 'echo'` 动作，逐个调用 `deleteCustom(id)`（确保 `caUnregister` 注销 BC 内 Activity）；
2. 清空 `state.echoSuppressed` 集合（或重导后由 `rebuildEchoSuppressed` 重建）；
3. 不碰 `source === 'native'` 的动作。
现有代码**只有单条 `deleteCustom`，没有批量删 echo 的函数**，需要新写且必须正确串联上述逻辑。

---

## 3. 更优替代方案

| 方案 | 做法 | 优点 | 缺点 |
|---|---|---|---|
| **D. 增量同步（推荐）** | 导入时计算 echo 当前动作集合，与现有 `source==='echo'` 对比：echo 已移除的→删除孤儿；存在的→更新；新增的→新增 | 精确、无感，保留用户编辑与排序，真正消除 C 类重复 | 实现稍复杂，需 diff 逻辑 |
| **B-opt. 逐条覆盖提示** | 导入时若发现某 echo 动作已被本地编辑过（dialog 与 echo 当前值不同），弹"保留我的 / 用 echo 的 / 跳过" | 信息透明、用户有控制权，解决 B 类丢编辑 | 交互步骤变多；echo 动作多时 prompt 疲劳 |
| **C-opt. 备份后删除** | 执行"清空重导"前，先自动 `exportCustomActions()` 下载一份 JSON 备份 | 给"全删"方案上了一道安全网 | 仍丢本地编辑，只是可恢复；多一步下载 |
| **现有覆盖更新（基线）** | 维持现状 | 零实现成本 | 不解决 B、C |

**建议组合**：以 **D（增量同步）** 为主，导入前做一次**静默自动备份**（写进 localStorage 而非强制下载，提供"上一次导入前快照"恢复入口）兜底。这比"全删重导"体验好得多，且不增加用户决策负担。

---

## 4. 用户体验考量

- **一刀切删除 + 重导**：用户每次导入都感觉"我的改动被重置"，容易产生"这插件会吞我的设置"的不信任感；且确认弹窗若默认"是"，一键就清掉全部。
- **增量同步（D）**：用户无感，导入永远是"最新状态 + 我的编辑都在"，最符合"导入 = 同步"的心智模型。
- **覆盖提示（B-opt）**：适合本地重度编辑用户，但 prompt 频繁会烦；适合做成"仅在检测到本地编辑差异时才问"。
- **备份后删（C-opt）**：心理安全，但属于"事后补救"，不如从源头避免。

---

## 5. 可行性结论

- **技术可行性**：✅ 可行。`importCustomFromEcho` 前加一步"若有 `source==='echo'` 数据则弹确认 → 批量 `deleteCustom` → 继续导入"即可。
- **是否为最优解**：❌ 否。它是"用最省事的方式解决重复"，但：
  1. 会误删被同化为 echo 的 native 动作（高危）；
  2. 消灭所有本地编辑与排序，不可恢复；
  3. 对真正的痛点（B 类本地编辑丢失）毫无改善。
- **推荐路径**：实现 **增量同步（D）+ 导入前自动快照备份**，替代"清空重导"。若坚持做"清空确认"按钮，至少要做到：
  1. 删除前自动导出备份 JSON；
  2. 删除时严格只处理 `source==='echo'` 且 `id` 由本插件生成（不碰被同化的 native，修复 2.1 隐患需要先把 `importCustomFromEcho` 第 51 行的 `existing.source='echo'` 改成"native 不动，仅 echo 才更新");
  3. 同步清空 `echoSuppressed` 并对每个动作 `caUnregister`。

### 利弊对比速览

| 维度 | 全删重导（用户方案） | 增量同步（推荐） |
|---|---|---|
| 解决重复(C) | ✅ | ✅ |
| 保留本地编辑(B) | ❌ 全丢 | ✅ 保留 |
| 误删 native 风险 | ⚠️ 高危 | ✅ 无 |
| 数据可恢复 | ❌（除非加备份） | ✅（快照） |
| 实现复杂度 | 低 | 中 |
| 用户心智 | 困惑"我的设置呢" | 无感同步 |
